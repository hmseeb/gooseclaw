  <action>
The apply_config() function (around line 815) has a generic loop `for env_var in env_map.get(provider_type, [])` that sets env vars using `config.get(env_var.lower(), "") or api_key`. This works for single-key providers but has issues for:

1. **azure-openai**: setup.json stores 'azure_key' and 'azure_endpoint', but env_map expects AZURE_OPENAI_API_KEY and AZURE_OPENAI_ENDPOINT. The current `config.get(env_var.lower())` looks for 'azure_openai_api_key' which doesn't exist in setup.json.
2. **litellm**: setup.json stores config as 'api_key' and host info, but env_map expects LITELLM_API_KEY and LITELLM_HOST.
3. **github-copilot**: env_map has GITHUB_TOKEN but setup.json stores 'api_key'.

Add a field_map dict in apply_config that maps env var names to their setup.json field names for providers with non-standard field naming:
```python
field_map = {
    'AZURE_OPENAI_API_KEY': 'azure_key',
    'AZURE_OPENAI_ENDPOINT': 'azure_endpoint',
    'LITELLM_HOST': 'litellm_host',
    'OLLAMA_HOST': 'ollama_host',
}
```

Update the generic env var loop to check field_map first:
```python
for env_var in env_map.get(provider_type, []):
    mapped_field = field_map.get(env_var, env_var.lower())
    val = config.get(mapped_field, "") or api_key
    if val:
        os.environ[env_var] = val
```

This preserves backward compatibility (field_map is a targeted override, not a rewrite).

Also verify that `default_models` dict has sensible values for all 23 providers (it currently does -- just confirm no stale model names).
