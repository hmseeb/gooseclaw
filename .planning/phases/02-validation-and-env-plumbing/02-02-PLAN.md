Implement credential validation gating in setup.html:

**1. Validation State Tracking:**
Add a global variable `let validationPassed = false;` near the other globals (selectedProvider, isReconfigure, etc.). Reset it to false whenever the provider changes (in selectProvider function).

**2. Per-Provider Format Validation (client-side):**
Add a `getFormatHint(providerId)` function that returns format hints for each provider:
- anthropic: "Starts with 'sk-ant-'"
- openai: "Starts with 'sk-'"
- google: "Starts with 'AI'"
- groq: "Starts with 'gsk_'"
- openrouter: "Starts with 'sk-or-'"
- mistral: "No specific format"
- xai: "Starts with 'xai-'"
- deepseek: "Starts with 'sk-'"
- together: "No specific format"
- cerebras: "No specific format"
- perplexity: "Starts with 'pplx-'"
- avian: "Starts with 'avian-'"
- venice: "No specific format"
- ovhcloud: "Token from OVH AI Endpoints dashboard"
- litellm: "Optional -- depends on your proxy config"
- claude-code: "Run `claude setup-token` locally"
- github-copilot: "No key needed"
- ollama/lm-studio/docker-model-runner/ramalama: "URL format (http://...)"

Add a `validateFormat(providerId, value)` function that returns {valid: boolean, error: string}:
- If provider is local (ollama, lm-studio, docker-model-runner, ramalama): validate URL format (starts with http:// or https://)
- If provider is claude-code: value must be non-empty
- If provider is github-copilot: always valid (no key)
- If provider is azure-openai: both key and endpoint must be non-empty, endpoint starts with https://
- For API key providers with known prefixes: check prefix. If prefix doesn't match, return warning (not hard error) -- `{valid: true, warning: "Key doesn't match expected format"}`
- For telegram token: must match pattern `/^\d+:[A-Za-z0-9_-]+$/`

**3. Display Format Hints:**
In `buildCredFields()`, after each API key input field, add a `<div class="field-hint format-hint">${getFormatHint(selectedProvider)}</div>`.

**4. Empty Field Blocking:**
In the "Continue" button handler for step 1 (credentials step), check:
- If provider requires a key (not github-copilot, not local without host requirement): verify the key input is non-empty
- If empty, show alert('error', 'API key is required') and don't advance
- Run `validateFormat()` and show any warnings/errors

**5. Validation Gate on Save:**
Modify the flow so the Save button on step 4 (confirmation) is initially disabled. Add text below it: "Validate your credentials on step 2 first."

Track validation state:
- After successful `validateKey()` (data.valid === true), set `validationPassed = true`
- For providers with `skip_validation: true` in the response (claude-code, github-copilot), auto-set `validationPassed = true`
- For local providers with no host configured, auto-set `validationPassed = true`

In step 4 (confirmation), if `validationPassed` is true, enable the Save button. If false, show a note: "Test your credentials before saving (go back to step 2)."

**6. Telegram Format Validation:**
In the Settings step (step 3), when the telegram token field loses focus, validate its format using the regex `/^\d+:[A-Za-z0-9_-]+$/`. If invalid and non-empty, show inline error text below the field.

<task type="auto">
  <name>Task 2: Pre-fill wizard fields on reconfigure</name>
  <files>docker/setup.html</files>
  <action>
When a user clicks "Reconfigure" from the dashboard (which calls a function to switch to the wizard flow), the wizard should pre-fill form fields with existing config values:

**1. Add a `prefillWizard(config)` function:**
This function receives the existing config object (from `/api/setup/config` response, already stored in `dashboardConfig`).

It should:
- Set `selectedProvider = config.provider_type` and call `selectProvider(config.provider_type)` to highlight the card
- After `buildCredFields()` runs (triggered by selectProvider), pre-fill credential inputs:
  - For standard API key providers: if `config.api_key_set` is true, set the input placeholder to "Key already set (leave blank to keep)" and DON'T fill the value (it's masked "********")
  - For claude-code: if `config.claude_setup_token_set` is true, set placeholder
  - For azure-openai: pre-fill endpoint from `config.azure_endpoint` (not secret), set key placeholder
  - For local providers: pre-fill host URL from `config.ollama_host`
  - For custom: pre-fill URL from `config.custom_url`, set key placeholder
- Pre-fill model field: `document.getElementById('model').value = config.model || ''`
- Pre-fill timezone: select the matching option in the timezone dropdown
- Pre-fill telegram token: if `config.telegram_bot_token_set`, set placeholder "Token already set"
- Set `validationPassed = true` (existing config was previously validated)

**2. Wire up the Reconfigure button:**
The dashboard's "Reconfigure" link/button (find it in the dashboard step HTML) should call:
```javascript
function startReconfigure() {
  document.getElementById('step-dashboard').classList.remove('active');
  document.getElementById('progressBar').style.display = '';
  document.getElementById('reconfigBanner').classList.add('show');
  prefillWizard(dashboardConfig);
  goToStep(0);
}
```

**3. Handle "keep existing" on save:**
In `saveConfig()`, when a credential field is empty AND the provider was previously configured (isReconfigure === true), don't send an empty string -- instead omit the field from the config payload so the backend keeps the existing value. Add logic:
- If `isReconfigure && !config.api_key && dashboardConfig.api_key_set`: delete config.api_key (keep existing)
- Same pattern for claude_setup_token, custom_key, azure_key, telegram_bot_token, web_auth_token

**4. Pre-fill on initial wizard load for reconfigure:**
The existing code at line 1543 (`fetch('/api/setup/config')`) already detects reconfigure and shows the dashboard. When the user navigates FROM dashboard to wizard, `prefillWizard` handles pre-fill. No changes needed to the initial load flow.
