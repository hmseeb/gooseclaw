  <name>Task 1: Fix CORS wildcard and add first-boot API lockdown</name>
  <files>docker/gateway.py</files>
  <action>
1. **Replace CORS wildcard with origin-aware header.** In `send_json()` (line ~1141), replace the hardcoded `Access-Control-Allow-Origin: *` with logic that:
   - Reads the `Origin` header from the request
   - If the request has no Origin header (same-origin or server-to-server), omit CORS header entirely
   - If the Origin matches the Host header's scheme+host (self-referential), echo it back
   - Otherwise, do NOT set any CORS header (browser will block cross-origin)
   - Also handle `do_OPTIONS()` preflight: instead of proxying to goose, return 200 with appropriate CORS headers (Allow-Origin echoed from Origin if same-host, Allow-Methods: GET/POST/PUT/DELETE/PATCH/OPTIONS, Allow-Headers: Content-Type/Authorization, Max-Age: 86400). Only for /api/* paths.

2. **Lock down first-boot window.** Currently, between deploy and first setup, all /api/* endpoints are fully open. Fix:
   - In `handle_validate()`, add a check: if `load_setup()` returns None (first boot) AND no env-var provider is configured, allow validate (needed for setup). Otherwise require auth.
   - In `handle_get_config()`, already has auth check -- good.
   - In `handle_notify()`, `handle_notify_status()`, `handle_telegram_status()`, `handle_telegram_pair()`: if NOT configured (first boot), return 403 with `{"error": "agent not configured yet"}`. These endpoints make no sense before setup.
   - The key insight: during first boot, only `/setup`, `/api/setup/save`, `/api/setup/validate`, and `/api/setup/config` should be accessible. Everything else returns 403.

3. **Add helper method** `_is_first_boot()` that returns True when `load_setup()` is None AND no env-var provider is configured. Use this in the lockdown checks.

Do NOT change the existing auth token mechanism. Do NOT add rate limiting (that is Plan 04).
