  <files>.dockerignore, Dockerfile</files>
  <action>
1. **Create `.dockerignore`** at project root with:
   ```
   .git
   .gitignore
   .planning
   .agents
   .claude
   *.md
   !identity/*.md
   !VERSION
   LICENSE
   .env
   .env.*
   __pycache__
   *.pyc
   node_modules
   .vscode
   .idea
   ```
   Note: We exclude *.md but re-include identity/*.md (needed at runtime) and VERSION.

2. **Optimize Dockerfile:**
   - Add metadata labels after FROM: `LABEL maintainer="gooseclaw" description="GooseClaw personal AI agent" org.opencontainers.image.source="https://github.com/..." `
   - Replace `python3-yaml` apt package with a pip install: `RUN pip3 install --no-cache-dir PyYAML==6.0.2` (pinned version). This requires adding `python3-pip` to the apt install list OR using `python3 -m pip`. Actually, python3-yaml via apt is fine for stdlib-only — keep it but document the version. On reflection, the audit finding (item 33) is about pinning, not pip vs apt. Since we use apt, pin via `apt-get install -y python3-yaml=6.0-1` or just document. The simpler approach: create a `requirements.txt` with `PyYAML==6.0.2` and install via pip3. But this adds pip as a dependency. Trade-off: keep apt-based install, just document it. The real value is the other items.
   - Add `USER` directive: After the final RUN, add `RUN useradd -r -s /bin/false gooseclaw` and `USER gooseclaw`. BUT — goose web and entrypoint.sh may need root for apt-get (claude CLI install). Solution: create user but keep root as default, add comment explaining Railway's RAILWAY_RUN_UID=0. The Dockerfile should at minimum create a non-root user and document the trade-off. Use: `RUN groupadd -r gooseclaw && useradd -r -g gooseclaw -d /app -s /sbin/nologin gooseclaw`. Do NOT switch USER since entrypoint.sh needs root for claude CLI install. Add a comment: `# NOTE: Runs as root because entrypoint.sh may install claude CLI via apt. For non-claude-code deployments, override with --user gooseclaw`.
   - Fix shebang inconsistency (item 32): gateway.py has `#!/usr/bin/env python3` shebang but is invoked via `python3` in entrypoint.sh. This is fine (shebang is ignored when invoked with python3 directly). No change needed, but ensure the chmod +x in Dockerfile is consistent.
   - Add `HEALTHCHECK` instruction: `HEALTHCHECK --interval=30s --timeout=5s --retries=3 CMD curl -f http://localhost:8080/api/health || exit 1`

3. **Reorganize COPY for better layer caching:**
   - Copy scripts and identity first (change less often): `COPY docker/ /app/docker/` then `COPY scripts/ /app/scripts/` then `COPY identity/ /app/identity/` then `COPY VERSION /app/VERSION`
   - Instead of `COPY . /app/` which copies everything, use specific COPY instructions.

2. **Update Dockerfile** to install from requirements.txt instead of apt python3-yaml:
   - Remove `python3-yaml` from the apt-get install line
   - Add `python3-pip` to apt-get install (if not already present) OR use `python3 -m ensurepip`
   - Add: `COPY docker/requirements.txt /app/docker/requirements.txt`

   - Place this COPY+RUN before the main COPY for layer caching (requirements change rarely)

Note: `--break-system-packages` is needed on Ubuntu 22.04+ with PEP 668. Alternatively, keep apt python3-yaml and skip this task if the added complexity is not worth it. The key deliverable is that requirements.txt EXISTS as documentation even if apt is the install method.

Decision: Create requirements.txt for documentation and future use. In the Dockerfile, keep apt-get python3-yaml (simpler, no pip needed) but add a comment referencing requirements.txt for version tracking.
