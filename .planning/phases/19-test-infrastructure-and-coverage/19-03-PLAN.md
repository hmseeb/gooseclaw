**class TestSetupValidate:**
- `test_validate_missing_provider(live_gateway, auth_session)` - POST /api/setup/validate with empty body or missing provider field, assert 400
- `test_validate_returns_json(live_gateway, auth_session)` - POST /api/setup/validate with a provider config, assert response is JSON (validation may fail due to no real API key, but the endpoint should respond properly, not crash)

Create `docker/tests/test_jobs.py` with HTTP-level tests. All tests use `live_gateway` and `auth_session` fixtures.

First, read the gateway.py job endpoint handlers to understand:
- What JSON fields POST /api/jobs expects (name, type, schedule/duration, etc.)
- What GET /api/jobs returns (array of job objects with id, name, type, etc.)
- How DELETE /api/jobs/<id> works (path parsing)
- How POST /api/jobs/<id>/run works

**class TestJobList:**
- `test_list_jobs_returns_200(live_gateway, auth_session)` - GET /api/jobs with auth, assert 200 and JSON array
- `test_list_jobs_requires_auth(live_gateway)` - GET /api/jobs without auth, assert 401
- `test_list_jobs_empty_initially(live_gateway, auth_session)` - GET /api/jobs, assert response is empty list (or list with only default jobs if any exist)

**class TestJobCreate:**
- `test_create_reminder_job(live_gateway, auth_session)` - POST /api/jobs with a reminder-type job payload, assert 200/201 and response contains job ID
- `test_create_job_requires_auth(live_gateway)` - POST /api/jobs without auth, assert 401
- `test_create_job_missing_fields(live_gateway, auth_session)` - POST /api/jobs with empty body, assert 400

**class TestJobDelete:**
- `test_delete_existing_job(live_gateway, auth_session)` - Create a job first, then DELETE /api/jobs/<id> with auth, assert 200. Then list jobs and confirm it's gone.
- `test_delete_nonexistent_job(live_gateway, auth_session)` - DELETE /api/jobs/nonexistent-id, assert 404

**class TestJobRun:**
- `test_run_job_manually(live_gateway, auth_session)` - Create a job, then POST /api/jobs/<id>/run, assert 200 or appropriate response. The job may not actually execute (no goosed), but the endpoint should respond.

**class TestScheduleEndpoints:**
- `test_schedule_upcoming(live_gateway, auth_session)` - GET /api/schedule/upcoming, assert 200 and JSON
- `test_schedule_context(live_gateway, auth_session)` - GET /api/schedule/context, assert 200

**Important:** The gateway uses do_DELETE or handles DELETE in do_GET/do_POST. Check the actual implementation. If DELETE is handled via a different method, read the handler to understand the routing.

**Important:** Job creation payload varies by job type. Read the handler to get the exact field names. A simple reminder like {"type": "reminder", "message": "test", "duration": "5m"} is a good starting payload.
