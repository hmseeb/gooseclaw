<task type="auto">
  <name>Task 2: Create e2e container integration test</name>
  <files>docker/tests/test_e2e_container.py</files>
  <action>
Create the e2e test file that imports fixtures from conftest_e2e.py (pytest auto-discovers conftest files in same directory).

The file should contain a single test class `TestE2EContainer` with these test methods, executed in order via `@pytest.mark.order` or by naming them with numeric prefixes so they naturally sort:

1. `test_01_container_boots_and_health_returns_200(docker_container)`:
   - GET {base_url}/api/health
   - Assert 200
   - Assert response JSON has "service": "gooseclaw"
   - This proves the container built and booted successfully

2. `test_02_setup_wizard_saves_config(docker_container)`:
   - POST {base_url}/api/setup/save with JSON body:
     ```json
     {
       "provider_type": "openai",
       "api_key": "sk-fake-test-key-not-real",
       "web_auth_token": "e2e-test-password-123"
     }
     ```
   - Assert 200
   - Assert response JSON has "success": true
   - This proves the setup wizard flow completes via HTTP

3. `test_03_can_login_after_setup(docker_container)`:
   - POST {base_url}/api/auth/login with JSON body: `{"password": "e2e-test-password-123"}`
   - Assert 200
   - Assert response has Set-Cookie header
   - This proves auth works after setup

4. `test_04_health_shows_goosed_status_after_setup(docker_container)`:
   - Wait up to 30 seconds for goosed to start (poll /api/health every 3s)
   - GET {base_url}/api/health
   - Assert 200
   - Assert response JSON contains "goosed" key (the field exists, value may be "running" or "starting" or "error" depending on fake API key, but the field must exist proving goosed lifecycle was triggered)
   - Note: with a fake API key, goosed may fail to fully start. That's OK. The test proves the SYSTEM TRIED to start goosed after setup. Full goosed health would require a real API key.

Use `requests` library for HTTP calls (already in requirements-dev.txt).
Add a module-level pytest marker: `pytestmark = pytest.mark.e2e` so these tests can be selectively run with `pytest -m e2e`.
Add to the top of the file: `pytest.importorskip("requests")`.

Important: The tests must be independent enough that if goosed doesn't fully start (fake API key), the test still validates the container lifecycle and setup flow. Test 4 should assert the goosed KEY exists in health JSON, not that goosed is "running".
